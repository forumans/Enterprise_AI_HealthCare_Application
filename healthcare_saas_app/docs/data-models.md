# Data Models

All models use SQLAlchemy 2.0 async ORM with PostgreSQL. Primary keys are UUIDs auto-generated by the database. All timestamps are UTC.

---

## Entity Relationship Overview

```
tenants
  └── users (tenant_id)
        ├── doctors (user_id)
        │     ├── doctor_availability (doctor_id)
        │     └── appointments (doctor_id)
        │           ├── medical_records (appointment_id)
        │           │     └── prescriptions (medical_record_id)
        │           │           └── pharmacies (pharmacy_id)
        │           └── audit_logs (record_id)
        ├── patients (user_id)
        │     ├── appointments (patient_id)
        │     └── patient_documents (patient_id)
        └── admins (user_id)
```

All tables except `tenants` include `tenant_id` for data isolation. Most tables support **soft delete** via `deleted_at` / `deleted_by` columns.

## Tenant

**Table:** `tenants`

The root entity. Every other record belongs to a tenant. In a SaaS deployment each healthcare organisation is a separate tenant.

| Column | Type | Notes |
|---|---|---|
| `id` | UUID | Primary key |
| `name` | VARCHAR(255) | Tenant display name |
| `domain` | VARCHAR(255) | Optional unique domain |
| `created_at` | TIMESTAMP | Auto-set on insert |

---

## User

**Table:** `users`

Authentication entity. One user per person. The `role` determines which profile table (doctor / patient / admin) is also populated.

| Column | Type | Notes |
|---|---|---|
| `id` | UUID | Primary key |
| `tenant_id` | UUID | FK → tenants |
| `email` | VARCHAR(255) | Unique within tenant |
| `password_hash` | TEXT | PBKDF2-SHA256 hash |
| `role` | ENUM | `ADMIN`, `DOCTOR`, `PATIENT` |
| `is_active` | BOOLEAN | Default `true` |
| `created_at` | TIMESTAMP | Auto-set on insert |
| `deleted_at` | TIMESTAMP | Null = active (soft delete) |
| `deleted_by` | UUID | FK → users (who deleted) |

## DoctorAvailability

**Table:** `doctor_availability`

Time slots that a doctor has made available for booking. Each slot has a status that moves from `AVAILABLE` → `BOOKED` when an appointment is created.

| Column | Type | Notes |
|---|---|---|
| `id` | UUID | Primary key |
| `tenant_id` | UUID | FK → tenants |
| `doctor_id` | UUID | FK → doctors |
| `slot_time` | TIMESTAMP | Date and time of the slot |
| `status` | ENUM | `AVAILABLE`, `BOOKED`, `BLOCKED` |
| `block_reason` | TEXT | Optional note when status is BLOCKED |
| `created_at` | TIMESTAMP | Auto-set on insert |

**Slot statuses:**
- `AVAILABLE` — Patient can book this slot
- `BOOKED` — Linked to an appointment
- `BLOCKED` — Doctor marked as unavailable (e.g. holiday)

## Appointment

**Table:** `appointments`

The core scheduling entity. Links a patient to a doctor at a specific time.

| Column | Type | Notes |
|---|---|---|
| `id` | UUID | Primary key |
| `tenant_id` | UUID | FK → tenants |
| `doctor_id` | UUID | FK → doctors |
| `patient_id` | UUID | FK → patients |
| `slot_id` | UUID | FK → doctor_availability (optional) |
| `appointment_time` | TIMESTAMP | Scheduled date/time |
| `status` | ENUM | `SCHEDULED`, `CONFIRMED`, `CANCELLED`, `COMPLETED` |
| `notes` | TEXT | Patient notes at booking time |
| `created_at` | TIMESTAMP | Auto-set on insert |
| `deleted_at` | TIMESTAMP | Soft delete |
| `deleted_by` | UUID | FK → users |

**Status lifecycle:**
```
SCHEDULED → CONFIRMED → COMPLETED
          ↘ CANCELLED
```

## AuditLog

**Table:** `audit_logs`

Immutable record of every write operation (INSERT, UPDATE, DELETE) across the system. Used for compliance and change tracking. No soft delete — audit records are permanent.

| Column | Type | Notes |
|---|---|---|
| `id` | UUID | Primary key |
| `tenant_id` | UUID | FK → tenants |
| `table_name` | VARCHAR(100) | Which table was modified |
| `record_id` | UUID | Which record was modified |
| `action_type` | VARCHAR(20) | `INSERT`, `UPDATE`, `DELETE` |
| `old_data` | JSONB | Snapshot before change (null for INSERT) |
| `new_data` | JSONB | Snapshot after change (null for DELETE) |
| `performed_by` | UUID | FK → users (who made the change) |
| `performed_at` | TIMESTAMP | When the change occurred |

## Soft Delete Pattern

Most tables implement soft delete via the `SoftDeleteMixin`:

```python
deleted_at: datetime | None   # None means the record is active
deleted_by: UUID | None       # Who deleted it
```

All queries automatically filter out soft-deleted records using:
```python
Model.deleted_at.is_(None)
```

Hard deletes are never performed on business data. This preserves the full audit trail and allows restoration of accidentally deleted records.
